## Presets Rendering Bug Investigation (Why all presets look identical)

Assumptions/limits
- Static code review only; no runtime traces collected here. Treat all claims as unverified until you run the verification steps below.

### Root cause hypothesis
- Material swap and/or program cache prevents define/blend changes from taking effect at draw time.
  - The `Points` being rendered may not be using the newly created `prebakedMaterial` (multiple material paths exist). See `apps/cryptiq-mindmap-demo/app/components/PointCloudStage.tsx:2589–2635` and earlier paths at `714–729`.
  - Program cache equivalence: compiled program may be reused without the `USE_GAUSSIAN` define encoded in the cache key. While `defines.USE_GAUSSIAN` is set (`apps/.../dreamdust/DreamdustMaterial.ts:675–678, 698–701, 808–810`), adding a `customProgramCacheKey()` is the robust fix.
- Bloom may be disabled by guards, removing a key differentiator between presets. Guards: `bloomActive = !simEnabled && bloom && !noBloomOverride && (forceBloomOverride || bloomEligible)` (apps/.../PointCloudStage.tsx:1507–1512 vicinity).
- Hard-coded `uInkIntensity = 0.75` (apps/.../PointCloudStage.tsx:1115–1120) flattens visual variance across presets.

### Verification steps (add temporary logs)
1) Confirm the material instance bound to Points updates per preset
   - Log uuids after preset change:
     - `console.info('[preset] points.material.uuid', stagePointsRef.current?.material.uuid)`
     - `console.info('[preset] prebakedMaterial.uuid', prebakedMaterial?.uuid)`
     - Expect: both uuids match and change when switching presets.

2) Confirm defines/blend state at compile time
   - After setting blending/defines in `DreamdustMaterial.ts:702–712`, log:
     - `console.info('[preset]', { preset, blending: material.blending, depthTest: material.depthTest, defines: {...material.defines} })`
   - In `onBeforeCompile` (`DreamdustMaterial.ts:792–812`), log:
     - `console.info('[compile]', { hasGaussian: !!shader.defines?.USE_GAUSSIAN })`
   - Expect: C/D1/D2 → `hasGaussian: true`; B1/B2/D1/D2 → additive blending value `2`.

3) Prove define actually recompiles
   - In FRAGMENT, temporarily gate a green fill:
     - Before sprite calc (`DreamdustMaterial.ts:535`):
       ```glsl
       #ifdef USE_GAUSSIAN
       gl_FragColor = vec4(0.0,1.0,0.0,1.0);
       return;
       #endif
       ```
   - Toggle A → C. Expect entire cloud turns green in C; if not, define not effective or wrong material bound.

4) Prove bloom applies
   - Log the active UnrealBloomPass params right after applying preset settings; ensure the same pass instance is mutated, not discarded.

5) Search for hidden overrides
   - Grep for later mutations of `points.material` or `material.blending/defines` and disable them during tests.

### Fix recommendations
- Add a custom program cache key to force recompiles per kernel:
  - After material creation:
    ```ts
    material.customProgramCacheKey = () => (material.defines?.USE_GAUSSIAN ? 'gauss-1' : 'gauss-0')
    ```
- Harden R3F material swap:
  - Ensure the visible `<points>` always uses the intended material; add a key on points: `<points key={\`points:${aestheticPreset}\`}>`.
  - Dispose previous material on preset change (`useEffect` cleanup) to avoid silent reuse.
- Remove or gate the hard-coded `uInkIntensity = 0.75` (apps/.../PointCloudStage.tsx:1115–1120) so presets diverge.
- Bloom: mutate the same pass object on preset change; verify via logging.
- Optional: call `renderer.compile(scene, camera)` once after preset switch to prewarm and confirm new program.

### Alternative explanations (if above fails)
- DepthTest=false additive presets (B2/D2) may be hidden by draw order or background; test on a black background to exaggerate additive.
- Global tonemapping/ACES can compress highlights; A/B with tonemapping off to expose additive differences.
- Reveal freeze always on (forcing `baseReveal=1.0` at `DreamdustMaterial.ts:555–557`) can homogenize alpha; ensure normal reveal for testing.

### Quick sanity tests
- Preset C “green test” (define path) → should flip immediately if define is active.

#### 2. **Code Quality: Gaussian Kernel Safety Guard** ✅

**Location:** `DreamdustMaterial.ts:538`
**Good practice:** Gaussian sprite call includes safety clamp:
```glsl
sprite = dreamdustGaussianSprite(gl_PointCoord, max(0.1, uSpriteSharpness));
```

This prevents `exp(-r2 * 0)` which would always return 1.0 (solid circle). Implementation is defensive.

#### 3. **Reveal Gating Logic - Redundant Checks** ⚠️

**Location:** `DreamdustMaterial.ts:547-557`
**Issue:** Triple-checking `uReveal >= 0.999` in sequence:
```glsl
if (uReveal >= 0.999) {
  revealStrength = 1.0;  // Line 548
}
float w = 0.08;
if (uReveal >= 0.999) {
  w = 0.14;              // Line 552
}
// ... smoothstep ...
if (uReveal >= 0.999) {
  baseReveal = 1.0;      // Line 556
}
```

**Analysis:** While redundant, this is **intentional per implementation plan** (Step 1.0, Phase 6.4 "freeze reveal and gating"). The plan explicitly requested this to avoid "cellular voids" during screenshot capture. Not a bug, but could be optimized post-evaluation.

#### 4. **Type Safety: Missing Preset Validation** ⚠️

**Location:** `PointCloudStage.tsx:1532-1535`
**Issue:** Bloom settings lookup returns `undefined` if preset not in map, falls back to default:
```typescript
const preset = BLOOM_PRESET_SETTINGS[aestheticPreset]
return preset ?? DEFAULT_BLOOM_SETTINGS
```

**Analysis:** This is safe due to `??` operator, but TypeScript should guarantee all 7 presets exist in map. Current code allows runtime mismatch if new preset added to type but not to map.

**Recommendation:** Add type assertion or exhaustiveness check.

#### 5. **Shader Compilation: Both Kernels Always Compiled** 💡

**Location:** `DreamdustMaterial.ts:515-517`


**Analysis:** This is correct for `#define` switching - preprocessor selects which to call, but both must be available. Not inefficient; GPU compiler will optimize out unused code path. **Correct implementation.**
