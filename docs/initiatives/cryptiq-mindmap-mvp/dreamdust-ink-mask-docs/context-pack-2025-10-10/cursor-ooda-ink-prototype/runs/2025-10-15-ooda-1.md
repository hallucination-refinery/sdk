### Root Cause Hypothesis
The dummy 1×1 Float32 DataTexture created in `DreamdustMaterial.ts` default block is **executing during module load** (before THREE.js context exists), which may create an invalid texture handle or trigger a FloatType extension error. When the shader tries to validate, it fails because `uVelocity` is bound to a malformed texture.

Alternative: The shader itself may have a GLSL syntax error in the velocity sampling block I added, but FluidSim shaders compiled fine (they logged init), so it's more likely the material shader.

## Decide

### Minimal Change
**Lazy-initialize the dummy texture** inside `ensureUniform` or `makeDreamdustMaterial` rather than at module load. This ensures THREE context exists and extensions are available when creating the DataTexture.

Alternatively, **temporarily disable velocity sampling** by setting `uVelToNdc: 0.0` as the default (rollback gate) to confirm the shader compiles without the velocity path, then re-enable once we verify the texture fix.

**Decision: Try rollback first** (fastest validation). Set `uVelToNdc` default to `0.0` so the runtime gate skips velocity sampling entirely. If scene renders, we confirm the issue is in texture initialization; if still black, the shader has a syntax error.

### Expected Impact
- If successful: scene renders; point cloud visible; no shader errors
- If still broken: shader syntax error in velocity block; need to audit GLSL
